Mandatory Chunk
Workflow Rules

These rules exist to prevent broken stitched files, duplication, and abbreviation issues that have repeatedly damaged development velocity.

Core Rule

MiniSimmy2 development uses a strict 3-chunk system. All work must follow this workflow without exception.

1
Always ask for the master known good complete file first

Before doing any analysis or editing, ask the user to provide (or confirm) the current master known good stitched file. Do not proceed from memory or outdated versions.

2
Identify the chunk break points

Once you have the master file, locate the official chunk boundaries using the comment markers:

<!-- ========== CHUNK 1 START ========== -->
<!-- ========== CHUNK 2 START ========== -->
<!-- ========== CHUNK 3 START ========== -->
3
Make all changes inside the correct chunk(s)

Apply fixes, new features, or refactoring only within the relevant chunk. Do not create cross-chunk edits that cause duplication.

4
Always deliver complete, full chunks

When providing updated code:

  • Return the entire, literal content of the chunk(s) being changed.
  • Never use placeholders (..., add the rest here, assume previous code, "for brevity", etc.).
  • The chunk must be complete and ready to be directly copy-pasted for re-stitching.
5
Chunk Formatting Requirements (Mandatory)

Every chunk you deliver must follow this exact structure:

START OF CHUNK
<!--
========================================================================
CHUNK X of 3 — [Clear Description, e.g. HTML Structure + Modals]
========================================================================
Changes in this version:
- Bullet list of what was added or fixed
========================================================================
-->
END OF CHUNK
<!-- ========== END OF CHUNK X ========== -->
6
No Duplication Policy

Never include code that already exists in another chunk. If code needs to be moved between chunks, explicitly remove it from the old chunk and place the clean version only in the new one.

7
Always Provide Re-stitching Instructions

When delivering chunks, include clear instructions such as:

“Replace your current Chunk 2 with this complete new Chunk 2.”
“Stitch in this order: Chunk 1 → Chunk 2 → Chunk 3”
Purpose of These Rules

These rules were created because repeated abbreviation, unclear boundaries, and duplication have caused broken stitched files multiple times. Following this workflow strictly prevents those problems and keeps development clean and efficient.
